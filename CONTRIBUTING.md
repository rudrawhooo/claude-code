# Contributing

Thanks for helping improve this project.

## Ground rules

- Be respectful and constructive.
- Keep changes focused and easy to review.
- Include tests or validation notes for behavior changes.
- Update docs when user-facing behavior changes.

## Legal and content policy

- Submit only code and content you have the right to contribute.
- Do not include proprietary, leaked, or confidential materials.
- Do not paste copyrighted closed-source code from other projects.
- If you reference external behavior, re-implement from public docs and clean-room reasoning.

## Development flow

1. Fork the repository.
2. Create a branch from `master`.
3. Implement and test your change.
4. Run formatting/linting/tests used by your local setup.
5. Open a pull request.

## Pull request checklist

- Clear title and summary
- Why this change is needed
- How it was tested
- Any breaking changes noted
- Docs updated (if needed)

## Commit style

Prefer clear commit messages, for example:

- `feat(commands): add XYZ behavior`
- `fix(bridge): handle timeout race`
- `docs: clarify setup instructions`

## Reporting bugs

Please include:

- Expected behavior
- Actual behavior
- Reproduction steps
- OS and runtime details
- Logs or screenshots when relevant
